# __1\. Architecture Overview__

Waecan is built as a set of independent Rust crates communicating through well\-defined internal interfaces\. This modularity allows each subsystem to be developed, tested, and audited in isolation before integration\.

### __1\.1 Crate Structure__

waecan/

  Cargo\.toml                    \# workspace manifest

  crates/

    waecan\-core/               \# consensus rules, block validation, UTXO model

    waecan\-crypto/             \# ring signatures, stealth addresses, key derivation

    waecan\-p2p/                \# libp2p\-based peer discovery and message propagation

    waecan\-mempool/            \# transaction pool, fee validation, ordering

    waecan\-miner/              \# RandomX integration, block template construction

    waecan\-storage/            \# RocksDB persistence layer for chain \+ UTXO set

    waecan\-rpc/                \# JSON\-RPC 2\.0 server for wallet and tooling

    waecan\-wallet/             \# key management, tx construction, address generation

    waecan\-node/               \# binary: wires all crates together, CLI entry point

    waecan\-cli/                \# binary: command\-line wallet \(waecan\-cli\)

### __1\.2 Data Flow__

The node binary coordinates all subsystems via Tokio async channels\. The critical path for a new transaction is:

1. User constructs a transaction in waecan\-wallet and submits via RPC\.
2. waecan\-rpc validates the RPC call format and forwards to waecan\-mempool\.
3. waecan\-mempool validates: fee >= 0\.001 WAEC, ring signature valid, no double\-spend against current UTXO set\.
4. If valid, transaction is broadcast to peers via waecan\-p2p\.
5. waecan\-miner includes the transaction in the next block template\.
6. When block is mined, waecan\-core validates the full block and updates waecan\-storage\.
7. The 0\.001 WAEC fee output is marked as burned \(OP\_RETURN equivalent\) and never enters the UTXO set\.

# __2\. Cryptographic Layer \(waecan\-crypto\)__

This is the most critical crate\. Every cryptographic decision made here is permanent once the network launches\. The programmer must implement this crate first and audit it thoroughly before touching any other subsystem\.

## __2\.1 Key Derivation__

### __2\.1\.1 Master Seed__

Every Waecan wallet is derived from a 256\-bit master seed generated by a CSPRNG \(ChaCha20Rng from the rand\_chacha crate\)\. The seed is encoded as a 24\-word BIP\-39 mnemonic for human backup\.

// waecan\-crypto/src/keys\.rs

use rand\_chacha::ChaCha20Rng;

use rand::SeedableRng;

pub struct MasterSeed\(\[u8; 32\]\);

impl MasterSeed \{

    pub fn generate\(\) \-> Self \{

        let mut rng = ChaCha20Rng::from\_entropy\(\);

        let mut seed = \[0u8; 32\];

        rng\.fill\_bytes\(&mut seed\);

        MasterSeed\(seed\)

    \}

    pub fn to\_mnemonic\(&self\) \-> Mnemonic \{

        // BIP\-39: 24 words from 256\-bit entropy

        Mnemonic::from\_entropy\(&self\.0\)\.expect\("valid entropy"\)

    \}

\}

*⚠️  The view key allows a user to share read\-only access to their transaction history \(e\.g\., to an auditor\) without exposing spending capability\.*

## __2\.2 Stealth Addresses__

Stealth addresses ensure that every transaction output is sent to a one\-time public key that is unlinkable to the recipient's permanent address\. An observer watching the blockchain cannot determine which outputs belong to which address\.

### __2\.2\.1 Address Format__

A Waecan address encodes both the spend public key and the view public key:

pub struct WaecanAddress \{

    pub spend\_public: ed25519\_dalek::PublicKey,  // 32 bytes

    pub view\_public:  ed25519\_dalek::PublicKey,  // 32 bytes

\}

// Encoded as Bech32m: wae1 \+ bech32m\(spend\_pub || view\_pub\)

// Total length: 4 \(prefix\) \+ 1 \(separator\) \+ ~104 chars = ~109 chars

// Example: wae1q9x2r4k8m7n3p5q6r9s0t1u2v3w4x5y6z7a8b9c0\.\.\.

### __2\.2\.2 One\-Time Output Key Generation \(Sender Side\)__

// Sender generates a random scalar r \(the "tx secret"\)

// Sender computes: R = r \* G  \(G = Ed25519 base point\)

// R is embedded in the transaction as the "tx public key"

//

// One\-time output key P for recipient with view\_pub V and spend\_pub S:

//   P = H\_s\(r \* V\) \* G \+ S

//

// where H\_s\(\) is Keccak\-256 mapped to a scalar

pub fn compute\_output\_key\(

    tx\_secret: &Scalar,

    recipient\_view\_pub: &PublicKey,

    recipient\_spend\_pub: &PublicKey,

\) \-> \(PublicKey, PublicKey\) \{  // returns \(output\_key, tx\_public\_key R\)

    let R = tx\_secret \* G;

    let shared\_secret = tx\_secret \* recipient\_view\_pub;  // r \* V

    let hs = keccak256\_to\_scalar\(&shared\_secret\.compress\(\)\.to\_bytes\(\)\);

    let P = hs \* G \+ recipient\_spend\_pub;

    \(P, R\)

\}

### __2\.2\.3 Output Key Scanning \(Recipient Side\)__

// Recipient uses view\_private key v to scan each output:

//   P' = H\_s\(v \* R\) \* G \+ S

// If P' == P \(the output key in the transaction\), this output belongs to the recipient\.

pub fn scan\_output\(

    tx\_public\_key: &PublicKey,    // R from transaction

    view\_private: &SecretKey,     // v

    spend\_public: &PublicKey,     // S

    output\_key: &PublicKey,       // P claimed in tx

\) \-> bool \{

    let shared\_secret = view\_private \* tx\_public\_key;  // v \* R = v \* r \* G = r \* V

    let hs = keccak256\_to\_scalar\(&shared\_secret\.compress\(\)\.to\_bytes\(\)\);

    let expected\_P = hs \* G \+ spend\_public;

    expected\_P == \*output\_key

\}

## __2\.3 Ring Signatures \(MLSAG\)__

Ring signatures allow a sender to prove ownership of one of N public keys \(the "ring"\) without revealing which one\. Waecan enforces a minimum ring size of 11 at consensus level\. The implementation uses MLSAG \(Multilayered Linkable Spontaneous Anonymous Group signatures\), the same scheme used by Monero since RingCT\.

### __2\.3\.1 Core Structs__

pub struct RingMember \{

    pub output\_key: PublicKey,    // the one\-time public key of a real or decoy output

    pub key\_image:  Option<CompressedEdwardsY>, // only Some for the real member

\}

pub struct RingSignature \{

    pub key\_image: CompressedEdwardsY, // I = x \* H\_p\(P\) \-\- proves ownership without revealing x

    pub c\_0: Scalar,                   // initial challenge

    pub s: Vec<Scalar>,                // response scalars, one per ring member

\}

pub struct Ring \{

    pub members: Vec<RingMember>,      // min 11 members \(1 real \+ 10 decoys\)

\}

### __2\.3\.2 Key Image__

The key image I = x \* H\_p\(P\) is deterministic given the private spend key x and the output key P\. It is included in every transaction and stored in the key image set\. If a key image appears twice, the transaction is a double\-spend and rejected at consensus\.

pub fn compute\_key\_image\(spend\_private: &SecretKey, output\_key: &PublicKey\) \-> CompressedEdwardsY \{

    // H\_p\(P\) = Keccak\-256\(P\) mapped to Ed25519 curve point

    let hp = hash\_to\_point\(output\_key\);

    let image = spend\_private \* hp;

    image\.compress\(\)

\}

### __2\.3\.3 Decoy Selection__

Decoys are selected from the global UTXO set using a triangular distribution biased toward recent outputs\. This mimics Monero's approach and prevents statistical attacks on ring composition\. The decoy selection algorithm must be identical for all wallet implementations to prevent fingerprinting\.

pub fn select\_decoys\(

    utxo\_set: &UtxoSet,

    real\_output: &OutputKey,

    count: usize,            // must be >= 10

    rng: &mut ChaCha20Rng,

\) \-> Vec<OutputKey> \{

    // 1\. Get total UTXO count N

    // 2\. For each decoy slot, sample index i from triangular\_distribution\(0, N, N\)

    //    \(biased toward N means biased toward recent outputs\)

    // 3\. Fetch UTXO at index i; if it equals real\_output, resample

    // 4\. Shuffle final ring to randomize real member position

\}

## __2\.4 Confidential Amounts \(Pedersen Commitments\)__

Transaction amounts are hidden using Pedersen commitments\. The sender commits to the amount without revealing it\. Validators can verify that inputs equal outputs \(no inflation\) without knowing the actual values\.

// Commitment: C = amount \* G \+ blinding\_factor \* H

// G and H are independent Ed25519 generator points

// H = H\_p\(G\) \-\- hash\-to\-point of G, no known discrete log relationship

pub struct PedersenCommitment \{

    pub commitment: CompressedEdwardsY,  // C = v\*G \+ b\*H

    // blinding\_factor b is kept secret by sender; transmitted encrypted to recipient

\}

// Balance proof: sum\(input\_commitments\) == sum\(output\_commitments\) \+ fee\_commitment

// fee\_commitment is a public commitment to the 0\.001 WAEC fee \(b=0, v=0\.001\)

// This proof is verified by all nodes without knowing any individual amount

*⚠️  Range proofs \(Bulletproofs\) must be included to prove that all amounts are positive \(no negative amounts that would allow inflation\)\. Each output must include a Bulletproof\. Use the bulletproofs crate from dalek\-cryptography\.*

## __3\.2 Atomic Units__

All internal arithmetic uses atomic units to avoid floating point\. 1 WAEC = 1,000,000,000,000 atomic units \(10^12\)\. This gives 12 decimal places of precision\. The minimum fee of 0\.001 WAEC = 1,000,000,000 atomic units\.

pub const ATOMIC\_UNITS\_PER\_WAEC: u64 = 1\_000\_000\_000\_000;  // 10^12

pub const MIN\_FEE\_ATOMIC:        u64 = 1\_000\_000\_000;       // 0\.001 WAEC

pub const MAX\_SUPPLY\_ATOMIC:     u128 = 105\_000\_000 \* ATOMIC\_UNITS\_PER\_WAEC as u128;

## __3\.3 Fee Burn Mechanism__

The fee is not added to the miner's coinbase reward\. It is destroyed\. This is the embryo of the Codex Ash Equation that will be expanded in V2\.

// During block validation, for each transaction:

// 1\. Verify fee >= MIN\_FEE\_ATOMIC

// 2\. Verify sum\(input\_commitments\) == sum\(output\_commitments\) \+ fee\_commitment

// 3\. The fee amount is NOT added to block\_reward; it is simply discarded

// 4\. A fee burn record is appended to the block's burn\_log: Vec<u64>

// 5\. Total burned WAEC is tracked in chain state for wallet display

// Block reward = base\_reward\(height\) ONLY

// Miners are NOT compensated by fees in V1

// This is intentional: fee burn aligns incentives toward WAEC scarcity

__❌  This design means miners are compensated solely by the block reward during V1\. This is only viable while the block reward is substantial \(early in the emission curve\)\. Fee compensation for miners must be revisited before halving cycle 4 \(~year 8\)\.__

## __3\.4 Transaction Validation Rules__

The following checks are performed in this exact order by waecan\-core::validate\_transaction\(\):

1. Version == 1\.
2. Number of inputs >= 1; number of outputs >= 1 and <= 16\.
3. Fee >= MIN\_FEE\_ATOMIC \(0\.001 WAEC = 1\_000\_000\_000 atomic units\)\.
4. For each input: ring size == 11 \(exactly; no more, no fewer in V1\)\.
5. For each input: key\_image is not in the global key image set \(double\-spend check\)\.
6. For each input: MLSAG ring signature is valid against the ring and the key image\.
7. For each output: Bulletproof range proof is valid\.
8. Balance proof: sum of input pseudo\-commitments == sum of output commitments \+ fee commitment\.
9. Transaction serialized size <= 100,000 bytes\.
10. extra field length <= 255 bytes\.
11. tx\_public\_key is a valid Ed25519 point \(not the identity, not low\-order\)\.

## __4\.2 Proof of Work — RandomX__

RandomX was chosen because it is memory\-hard and requires a full CPU to mine efficiently\. GPU and ASIC mining is economically unviable due to the algorithm's design\. This ensures decentralized mining by ordinary hardware\.

### __4\.2\.1 Rust Integration__

// Cargo\.toml dependency:

// randomx\-rs = "1\.2"  \(Rust bindings to the reference C\+\+ implementation\)

use randomx\_rs::\{RandomXCache, RandomXDataset, RandomXVM, RandomXFlag\};

pub fn compute\_pow\(header: &BlockHeader, seed\_hash: &\[u8; 32\]\) \-> \[u8; 32\] \{

    let flags = RandomXFlag::get\_recommended\_flags\(\);

    let cache  = RandomXCache::new\(flags, seed\_hash\)\.unwrap\(\);

    let vm     = RandomXVM::new\(flags, Some\(&cache\), None\)\.unwrap\(\);

    let input  = header\.serialize\(\);  // canonical serialization, see 4\.3

    vm\.calculate\_hash\(&input\)\.unwrap\(\)\.try\_into\(\)\.unwrap\(\)

\}

// Seed hash changes every 2048 blocks \(same interval as Monero\)

// seed\_hash at height H = block\_hash\(H \- \(H mod 2048\)\)

// This allows miners to pre\-compute the RandomX dataset

pub fn get\_seed\_hash\(height: u64, chain: &Chain\) \-> \[u8; 32\] \{

    let seed\_height = height \- \(height % 2048\);

    chain\.block\_hash\_at\(seed\_height\)

\}

### __4\.2\.2 Block Validity Check__

pub fn is\_pow\_valid\(header: &BlockHeader, pow\_hash: &\[u8; 32\]\) \-> bool \{

    // pow\_hash as 256\-bit little\-endian integer must be <= target

    let hash\_int = U256::from\_le\_bytes\(\*pow\_hash\);

    let target   = compact\_to\_target\(header\.difficulty\);

    hash\_int <= target

\}

## __4\.3 Canonical Serialization__

Every struct that is hashed must have a canonical \(deterministic\) serialization\. Use bincode with little\-endian, fixed\-width integers\. Never hash a Debug or Display representation\. The serialization format is:

// BlockHeader serialization order \(for PoW input and for block hash\):

// \[0\.\.1\]   version:    u8

// \[1\.\.33\]  prev\_hash:  \[u8; 32\]

// \[33\.\.65\] merkle\_root:\[u8; 32\]

// \[65\.\.73\] timestamp:  u64 little\-endian

// \[73\.\.81\] difficulty: u64 little\-endian

// \[81\.\.89\] nonce:      u64 little\-endian

// \[89\.\.97\] height:     u64 little\-endian

// Total: 97 bytes

pub fn serialize\_header\(h: &BlockHeader\) \-> \[u8; 97\] \{

    let mut buf = \[0u8; 97\];

    buf\[0\] = h\.version;

    buf\[1\.\.33\]\.copy\_from\_slice\(&h\.prev\_hash\);

    buf\[33\.\.65\]\.copy\_from\_slice\(&h\.merkle\_root\);

    buf\[65\.\.73\]\.copy\_from\_slice\(&h\.timestamp\.to\_le\_bytes\(\)\);

    buf\[73\.\.81\]\.copy\_from\_slice\(&h\.difficulty\.to\_le\_bytes\(\)\);

    buf\[81\.\.89\]\.copy\_from\_slice\(&h\.nonce\.to\_le\_bytes\(\)\);

    buf\[89\.\.97\]\.copy\_from\_slice\(&h\.height\.to\_le\_bytes\(\)\);

    buf

\}

## __4\.4 Difficulty Adjustment — ASERT\-DAA__

ASERT \(Absolutely Scheduled Exponentially Rising Targets\) provides smooth, responsive difficulty adjustment that prevents block\-time oscillation under variable hashrate\. It was designed by Mark Lundeberg and is used in production on Bitcoin Cash\.

// ASERT\-DAA constants for Waecan:

pub const BLOCK\_TIME\_TARGET:   i64 = 120;       // 120 seconds = 2 minutes

pub const ASERT\_HALFLIFE:      i64 = 86\_400;    // 1 day in seconds

// Anchor: block 0 \(genesis\); anchor\_bits = initial\_difficulty

pub fn asert\_next\_target\(

    anchor\_bits:      u64,    // difficulty of anchor block

    anchor\_height:    u64,

    anchor\_timestamp: u64,

    prev\_height:      u64,

    prev\_timestamp:   u64,

\) \-> u64 \{

    let height\_diff   = \(prev\_height \- anchor\_height\) as i64;

    let time\_diff     = prev\_timestamp as i64 \- anchor\_timestamp as i64;

    let exponent      = \(time\_diff \- BLOCK\_TIME\_TARGET \* height\_diff\) \* \(1 << 16\)

                        / ASERT\_HALFLIFE;

    // new\_target = anchor\_target \* 2^\(exponent / 65536\)

    // Implemented as 128\-bit integer arithmetic to avoid float

    asert\_compute\_target\(anchor\_bits, exponent\)

\}

*⚠️  ASERT recalculates difficulty on every block, not every 720 blocks as originally planned\. This is strictly better: smoother adjustment, no oscillation window for timestamp manipulation\. The 720\-block anchor update interval determines when the anchor reference point shifts forward\.*

## __5\.4 Message Propagation Rules__

To prevent DoS and spam propagation, the following rules apply before any message is forwarded:

- Blocks: must pass full validation \(Section 4 \+ Section 3\)\. Invalid blocks are rejected and the peer's reputation is decremented\.
- Transactions: must pass transaction validation \(Section 3\.4\)\. Fee < MIN\_FEE is rejected silently \(not penalized; could be a version mismatch\)\.
- Message deduplication: maintain a 10,000\-entry LRU cache of recent message hashes\. If seen, discard without forwarding\.
- Max inbound connections: 125\. Max outbound: 8\. Same as Bitcoin Core defaults\.
- Ban list: peers sending > 3 invalid blocks within 60 seconds are banned for 24 hours\.

## __6\.3 Atomic Block Commit__

Block storage and UTXO set updates must be atomic\. Use RocksDB WriteBatch:

pub fn commit\_block\(db: &DB, block: &Block, new\_outputs: Vec<OutputRecord>,

                    spent\_keys: Vec<PublicKey>, key\_images: Vec<CompressedEdwardsY>\) \{

    let mut batch = WriteBatch::default\(\);

    // 1\. Store block

    batch\.put\_cf\(CF\_BLOCKS, &block\.header\.hash\(\), &block\.serialize\(\)\);

    // 2\. Update UTXO set: add new outputs

    for out in &new\_outputs \{

        batch\.put\_cf\(CF\_UTXO, &out\.output\_key\.compress\(\)\.as\_bytes\(\), &out\.serialize\(\)\);

    \}

    // 3\. Update UTXO set: remove spent outputs

    for key in &spent\_keys \{

        batch\.delete\_cf\(CF\_UTXO, &key\.compress\(\)\.as\_bytes\(\)\);

    \}

    // 4\. Record key images \(double\-spend prevention\)

    for img in &key\_images \{

        batch\.put\_cf\(CF\_KEY\_IMAGES, img\.as\_bytes\(\), &block\.header\.height\.to\_le\_bytes\(\)\);

    \}

    // 5\. Update chain tip

    batch\.put\_cf\(CF\_CHAIN\_META, b"tip\_hash",   &block\.header\.hash\(\)\);

    batch\.put\_cf\(CF\_CHAIN\_META, b"tip\_height", &block\.header\.height\.to\_le\_bytes\(\)\);

    db\.write\(batch\)\.expect\("atomic block commit failed"\);  // panics if RocksDB fails

\}

pub fn build\_block\_template\(chain: &Chain, mempool: &Mempool, miner\_addr: &WaecanAddress\)

    \-> BlockTemplate \{

    // 2\. Build coinbase with miner's stealth output for block\_reward\(chain\.height \+ 1\)

    // 3\. Compute Merkle root over \[coinbase\_tx\_hash, tx\_1\_hash, \.\.\., tx\_n\_hash\]

## __8\.2 Mining Loop__

pub fn mine\(template: BlockTemplate, seed\_hash: \[u8; 32\]\) \-> Block \{

    let flags  = RandomXFlag::get\_recommended\_flags\(\);

    let cache  = RandomXCache::new\(flags, &seed\_hash\)\.unwrap\(\);

    let vm     = RandomXVM::new\(flags, Some\(&cache\), None\)\.unwrap\(\);

    let target = template\.target;

    let mut header = template\.header\.clone\(\);

    let mut nonce: u64 = 0;

    loop \{

        header\.nonce = nonce;

        let hash = vm\.calculate\_hash\(&serialize\_header\(&header\)\)\.unwrap\(\);

        if U256::from\_le\_slice\(&hash\) <= U256::from\_le\_bytes\(target\) \{

            return Block \{ header, coinbase: template\.coinbase, transactions: template\.transactions \};

        \}

        nonce = nonce\.wrapping\_add\(1\);

        if nonce == 0 \{

            // nonce exhausted; update timestamp to get fresh PoW space

            header\.timestamp = current\_unix\_time\(\);

        \}

    \}

\}

*⚠️  For multi\-threaded mining, spawn N threads, each starting at nonce = thread\_id, incrementing by N\. Share a stop flag \(AtomicBool\) so all threads stop when one finds a solution\.*

## __9\.3 Blockchain Scanning__

The wallet scans every transaction output in every block using the view private key\. This is the standard Monero\-style scanning approach\. For V1, the wallet does a full chain scan on first load and caches results\.

pub fn scan\_block\(block: &Block, view\_private: &SecretKey, spend\_public: &PublicKey\)

    \-> Vec<OwnedOutput> \{

    let mut owned = Vec::new\(\);

    for tx in &block\.transactions \{

        let R = tx\.tx\_public\_key;

        for \(i, out\) in tx\.outputs\.iter\(\)\.enumerate\(\) \{

            if scan\_output\(&R, view\_private, spend\_public, &out\.output\_key\) \{

                let amount = decrypt\_amount\(&out\.encrypted\_amount, view\_private, &R\);

                owned\.push\(OwnedOutput \{ tx\_hash: tx\.hash\(\), output\_index: i as u8,

                    output\_key: out\.output\_key, amount \}\);

            \}

        \}

    \}

    owned

\}

# __10\. QR Payment Protocol__

The QR payment protocol enables the Waecan wallet to function as a peer\-to\-peer payment system identical in user experience to PIX, but with full cryptographic privacy\.

## __10\.1 QR Payload Format__

A payment request is encoded as a URI, then represented as a QR code\. The URI format:

waecan:<address>?amount=<atomic\_units>&nonce=<hex16>&exp=<unix\_ts>&note=<urlencoded>

// Example:

// waecan:wae1q9x2r4k8m7n3p5\.\.\.?amount=850000000000&nonce=a3f9c21b8e047d5c&exp=1735900800&note=Caf%C3%A9\+duplo

pub struct PaymentRequest \{

    pub address:  WaecanAddress,   // recipient's permanent address

    pub amount:   u64,             // in atomic units

    pub nonce:    \[u8; 8\],         // random; makes each QR unique

    pub expiry:   u64,             // Unix timestamp; reject if current\_time > expiry

    pub note:     Option<String>,  // max 64 bytes UTF\-8; stored in tx\.extra

\}

## __10\.2 QR Generation \(Merchant Side\)__

pub fn generate\_payment\_qr\(

    address: &WaecanAddress,

    amount\_waec: f64,

    note: Option<&str>,

    ttl\_seconds: u64,    // default: 600 \(10 minutes\)

\) \-> QrPayload \{

    let nonce: \[u8; 8\] = random\_bytes\(\);

    let expiry = current\_unix\_time\(\) \+ ttl\_seconds;

    let amount\_atomic = \(amount\_waec \* ATOMIC\_UNITS\_PER\_WAEC as f64\) as u64;

    let uri = format\!\("waecan:\{\}?amount=\{\}&nonce=\{\}&exp=\{\}&note=\{\}",

        address\.to\_bech32m\(\),

        amount\_atomic,

        hex::encode\(nonce\),

        expiry,

        urlencoding::encode\(note\.unwrap\_or\(""\)\)

    \);

    QrPayload \{ uri, nonce, expiry \}

\}

// Each call generates a cryptographically unique QR\.

// Two QRs for the same amount are unlinkable on the blockchain\.

## __10\.3 QR Scanning and Payment \(Payer Side\)__

pub fn process\_scanned\_qr\(raw\_qr: &str, wallet: &Wallet, utxo\_set: &\[UnspentOutput\]\)

    \-> Result<Transaction, PaymentError> \{

    // 1\. Parse URI and decode PaymentRequest

    let req = PaymentRequest::from\_uri\(raw\_qr\)?;

    // 2\. Check expiry

    if current\_unix\_time\(\) > req\.expiry \{

        return Err\(PaymentError::QrExpired\);

    \}

    // 3\. Check payer has sufficient balance

    let total\_needed = req\.amount \+ MIN\_FEE\_ATOMIC;

    if wallet\.balance\(\) < total\_needed \{

        return Err\(PaymentError::InsufficientFunds \{ have: wallet\.balance\(\), need: total\_needed \}\);

    \}

    // 4\. Build and sign transaction

    let tx = build\_transaction\(wallet, &req\.address, req\.amount, utxo\_set, &mut rng\)?;

    // 5\. Return signed transaction \(caller broadcasts via RPC\)

    Ok\(tx\)

\}

# __14\. Security Considerations & Adversary Model__

This section lists known attack vectors and the protocol's response to each\. Every item here must be verified by the programmer before mainnet launch\.

## __14\.1 Cryptographic Attacks__

- Spend key extraction from ring signature: MLSAG is computationally binding under DLEQ hardness assumption on Ed25519\. No known polynomial\-time break\.
- Stealth address linkability: outputs are unlinkable unless the attacker holds the view private key\. Even the transaction sender cannot link two payments to the same recipient without the view key\.
- Commitment opening \(amount reveal\): Pedersen commitments are computationally hiding under discrete log hardness\. Bulletproofs are zero\-knowledge\.
- Key image forgery: I = x \* H\_p\(P\) is binding because H\_p maps to a point with unknown discrete log relative to G\. Forging a key image without knowing x requires solving DLEQ, which is hard\.

## __14\.2 Network Attacks__

- Sybil attack on peer discovery: Kademlia DHT has known Sybil resistance weaknesses\. Mitigate by requiring proof\-of\-work for peer ID registration \(V2\)\. In V1, rely on diverse seed node geography\.
- Eclipse attack: maintain minimum 8 outbound connections to peers selected from diverse /16 IP prefixes\. Reject inbound\-only configurations\.
- Transaction spam: minimum fee of 0\.001 WAEC makes large\-scale spam economically costly\. Mempool has a maximum size of 50MB; lowest\-fee transactions are evicted when full\.
- Block withholding: a miner who withholds a valid block loses the race to the next block\. No additional protocol defense needed in PoW\.

## __14\.3 Targeted Slash \(Addressed in V2\)__

As identified in the architectural review: a well\-funded adversary can DDoS specific nodes to trigger Slashing penalties in the Codex storage protocol\. This is a V2 concern\. The V1 protocol has no Slashing mechanism and is therefore not vulnerable to this attack\.

## __14\.4 Consensus Attacks__

- 51% attack: with low initial hashrate, a 51% attack is feasible\. Mitigation: launch with multiple mining pools and coordinate hashrate from the team\. Monitor for hashrate concentration > 40% and alert community\.
- Selfish mining: RandomX's memory\-hardness makes orphan races expensive\. Maintain standard 1\-block confirmation minimum; 10 blocks for significant transfers\.
- Timestamp manipulation: ASERT\-DAA is resistant to moderate timestamp manipulation\. Block timestamps must be > median of last 11 blocks and < network time \+ 2 hours\.
